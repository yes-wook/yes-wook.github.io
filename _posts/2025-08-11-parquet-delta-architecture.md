<h3>Parquet와 Delta Table: 배경과 아키텍처 이해하기</h3><blockquote>현재 Databricks 기반 분석 플랫폼을 운영하면서 쌓인 경험을 바탕으로, 기본이 되는 <strong>Parquet 파일</strong>과 <strong>Delta Table</strong>의 구조를 정리합니다.</blockquote><blockquote>이 글은 3편 시리즈 중 1편으로, 2편에서는 <strong>Deletion Vector</strong>, 3편에서는 <strong>Optimize/Z-Order/Reorg/Vacuum</strong> 다룰 예정입니다.</blockquote><h3>0. 들어가며</h3><p>데이터 엔지니어링 업무에서는 “데이터를 어떤 형식으로 저장할 것인가?”라는 질문을 자주 마주하게 됩니다.</p><p>특히 클라우드 환경에서 수십 테라바이트 이상의 데이터를 다룰 때는 저장 효율성과 읽기 성능을 동시에 만족시키는 포맷이 필요합니다.</p><p>이 글에서는 대규모 데이터 분석 환경에서 표준처럼 쓰이는<strong> Apache Parquet</strong>과, 이를 확장한<strong> Delta Table의 구조와 동작 원리</strong>를 설명합니다.</p><h3>1. Parquet의 등장 배경</h3><blockquote>Row-based vs Columnar Storage</blockquote><p>데이터 저장 포맷은 크게 <strong>행 기반(Row-based)</strong>과 <strong>컬럼 기반(Columnar)</strong>으로 나뉩니다.</p><ul><li><strong>Row-based 포맷</strong>: 한 행의 모든 컬럼을 연속적으로 저장 → 일반적으로, OLTP(트랜잭션 처리) 환경에 적합</li><li><strong>Columnar 포맷</strong>: 동일 컬럼 데이터를 묶어서 저장 → 일반적으로, 분석(OLAP) 환경에 최적화</li></ul><p>Row-based 포맷은 행 단위 처리에는 빠르지만, 특정 컬럼만 조회하는 분석 쿼리에서는 불필요한 데이터까지 읽어야 하는 단점이 있습니다.</p><p>반면 Columnar 포맷은 필요한 컬럼만 읽을 수 있어 I/O를 최소화하고, 압축 효율도 높습니다.</p><blockquote>Apache Parquet의 특징</blockquote><p>Parquet는 <strong>컬럼 기반 저장 포맷</strong>으로, 다음과 같은 장점을 갖습니다.</p><ol><li><strong>컬럼 단위 압축 : </strong>같은 컬럼 내 값들은 데이터 패턴이 유사해 압축 효율이 높음 → 스토리지 비용 절감, 네트워크 전송량 감소</li><li><strong>다양한 압축 알고리즘 지원<br>- </strong>Snappy (기본): 빠른 압축·해제, 적당한 압축률<br>- Gzip/ZSTD: 더 높은 압축률, 상대적으로 느림</li><li><strong>페이지 단위 데이터 구조 : </strong>Parquet는 컬럼을 페이지(Page) 단위로 나누어 저장하며, 페이지는 다시. <strong>데이터 페이지</strong>와 <strong>딕셔너리 페이지</strong>로 구성됩니다.</li></ol><h3>2. Parquet 파일 내부 구조</h3><p>Parquet 파일은 다음과 같은 계층 구조를 가집니다.</p><pre>+-----------------+<br>|   File Header   |<br>+-----------------+<br>|   Row Group 1   |<br>|  +-----------+  |<br>|  | Column 1  |  |<br>|  |  +-----+  |  |<br>|  |  |Page1|  |  |<br>|  |  |Page2|  |  |<br>|  |  +-----+  |  |<br>|  +-----------+  |<br>|  +-----------+  |<br>|  | Column 2  |  |<br>|  |  +-----+  |  |<br>|  |  |Page1|  |  |<br>|  |  |Page2|  |  |<br>|  |  +-----+  |  |<br>|  +-----------+  |<br>+-----------------+<br>|   Row Group 2   |<br>|    ...          |<br>+-----------------+<br>|   File Footer   |<br>+-----------------+</pre><ul><li><strong>Row Group</strong>: 데이터의 논리적 수평 분할 단위</li><li><strong>Column Chunk</strong>: 하나의 Row Group 내 특정 컬럼 데이터</li><li><strong>Page</strong>: Column Chunk를 이루는 최소 단위</li></ul><p><strong>구조 요약 :</strong></p><ul><li><strong>1 파일</strong> = 여러 개의 <strong>Row Group</strong></li><li><strong>1 Row Group</strong> = 여러 개의 <strong>Column Chunk</strong></li><li><strong>1 Column Chunk</strong> = 하나의 <strong>컬럼 데이터</strong></li><li><strong>1 Column Chunk</strong> = 여러 <strong>Page</strong></li></ul><p><strong>예시 파일명</strong>:</p><pre>part-00129-cf1d1014-4132-43df-9ef9-4095d639a8f7.c000.snappy.parquet</pre><ul><li>snappy: 기본 압축 라이브러리</li></ul><h3>3. Delta Lake와 Delta Table</h3><blockquote>왜 Delta Lake가 필요한가?</blockquote><p>Delta Lake는 Apache Parquet 기반의 <em>오픈소스 스토리지 레이어</em>입니다.<br>그리고 Delta Table은 이 Delta Lake 포맷을 사용해 <em>저장된 실제 테이블 인스턴스</em>를 의미합니다.</p><p>Parquet는 뛰어난 저장 효율성과 읽기 성능을 제공하지만, 트랜잭션 관리나 데이터 변경 이력 기능은 지원하지 않습니다.</p><p>따라서 대규모 분산 환경에서 안정적인 ACID 트랜잭션을 보장하려면, 이러한 기능을 제공하는 메타데이터 계층이 필요합니다.</p><blockquote>Delta Table = Parquet + Delta Log</blockquote><p>Delta Table은 Parquet 파일 위에 <strong><em>트랜잭션 로그 계층(_delta_log)</em></strong>을 추가하여 다음 기능을 제공합니다.</p><ul><li>ACID 트랜잭션</li><li>스키마 진화(Schema Evolution)</li><li>시간여행(Time Travel)</li><li>데이터 변경 이력 관리</li></ul><blockquote>Delta Table 디렉토리 예시:</blockquote><pre>/delta-table/<br>├── _delta_log/<br>│   ├── 00000000000000000000.json    # 최초 스냅샷<br>│   ├── 00000000000000000001.json    # 데이터 변경 로그 <br>│   ├── 00000000000000000002.checkpoint.parquet    # JSON 로그 병합본 생성<br>├── part-00000-xxxx.snappy.parquet<br>└── part-00001-xxxx.snappy.parquet</pre><h3>4._delta_log의 역할</h3><ul><li><strong>JSON 로그</strong>: 테이블 변경 사항(추가, 삭제, 스키마 변경 등) 기록</li><li><strong>Checkpoint Parquet</strong>: 여러 JSON 로그를 병합한 요약본 → 로딩 속도 향상</li><li><strong>데이터 파일(Parquet)</strong>: 실제 데이터 저장소</li></ul><p>이 구조 덕분에 Delta Table은 데이터 변경 시 전체 Parquet 파일을 재작성하지 않고도, 메타데이터만 수정하여 빠르게 상태를 반영할 수 있습니다.</p><h3>5. 마무리</h3><p>Parquet와 Delta Table은 현대 데이터 레이크·레이크하우스 아키텍처에서 핵심적인 역할을 수행합니다.</p><p>Parquet는 컬럼 기반 저장 구조로 성능과 압축 효율을 보장하고, Delta Table은 여기에 트랜잭션과 이력 관리 기능을 더해 안정적인 데이터 관리 환경을 제공합니다.</p><p>다음 글에서는 Delta Table의 <strong>Deletion Vector</strong> 개념과, 이를 활용한 효율적인 삭제 및 운영 전략을 다루겠습니다.</p><h4>참고자료</h4><ul><li><a href="https://delta.io/blog/2023-07-05-deletion-vectors/">https://delta.io/blog/2023-07-05-deletion-vectors/</a></li><li><a href="https://data-engineer-tech.tistory.com/52">https://data-engineer-tech.tistory.com/52</a></li><li><a href="https://parquet.apache.org/docs/overview/">https://parquet.apache.org/docs/overview/</a></li></ul><h4>참고코드</h4><p>아래와 같은 코드로, 파케이에서 Row Group 내용을 확인했습니다.</p><pre>import pyarrow.parquet as pq<br><br># Parquet 파일 경로<br>file_path = &quot;path/to/part-00129-...snappy.parquet&quot;<br><br># Parquet 파일 읽기<br>pf = pq.ParquetFile(file_path)<br><br># Row Group 정보 출력<br>for i in range(pf.num_row_groups):<br>    rg = pf.metadata.row_group(i)<br>    print(f&quot;Row Group {i}:&quot;)<br>    print(f&quot;  Length: {rg.total_byte_size}&quot;)<br>    print(f&quot;  Rows: {rg.num_rows}&quot;)</pre>
