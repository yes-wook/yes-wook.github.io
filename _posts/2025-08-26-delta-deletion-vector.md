<p>데이터 플랫폼을 운영하다 보면 <strong>데이터를 어떻게 삭제할 것인가?</strong> 하는 문제를 자주 맞닥뜨리게 됩니다.</p><p>단순히 레코드를 지우는 것 같지만, 분산 스토리지 환경에서는 생각보다 복잡한 과정이 필요합니다.</p><p>특히 Delta Table은 기존 Parquet 파일을 직접 덮어쓰지 않고,</p><p><strong>Deletion Vector(DV)</strong> 라는 개념을 통해 삭제를 효율적으로 처리합니다.</p><p>이번 글에서는 DV의 원리와 운영 전략을 살펴봅니다.</p><h3>1. 기존 Parquet 기반 삭제 방식의 한계</h3><ul><li>Parquet는 <strong>불변(immutable) 파일 포맷</strong>입니다.</li><li>데이터를 삭제하거나 업데이트 하려면, 원본 파일을 다시 써야 합니다.<br>→ <strong>Copy-on-Write(CoW)</strong> 전략</li></ul><pre>[ 원래 방식: Copy-on-Write(CoW) ]<br>Parquet File<br> ├── Row Group 1 (A,B,C,D)<br> └── Row Group 2 (E,F,G,H)<br><br>→ G 삭제 요청 → 새로운 Parquet 파일 재작성</pre><p>문제가 되는 부분:</p><ul><li>여러 파일이 얽히면, 삭제/업데이트 시 <strong>대량의 파일 재작성 필요</strong></li><li>스토리지 I/O 증가</li><li>처리 지연 → 운영 효율 저하</li></ul><h3>2. Deletion Vector란?</h3><p>Deletion Vector는 Delta Lake에서 도입된 새로운 방식으로,</p><p><strong>“삭제된 레코드를 표시만 하고 실제 파일은 건드리지 않는 기법”</strong>입니다.</p><ul><li>삭제 시: 대상 Row에 <strong>삭제 마커</strong>만 기록</li><li>쿼리 시: DV 참조 → 해당 Row 무시</li><li>실제 파일 재작성은 Optimize/Reorg 시점으로 지연</li></ul><pre>[ Deletion Vector 방식 ]<br>Parquet File<br> ├── Row Group 1 (A,B,C,D)<br> └── Row Group 2 (E,F,G*,H)<br><br>* G: DV로 삭제 표시됨 → 쿼리 시 무시됨</pre><h3>3. Delta Log와 DV 파일 예시</h3><p>삭제가 발생하면 Delta Table에는 <strong>두 가지 변화</strong>가 생깁니다.</p><ol><li>_delta_log JSON에 <strong>remove / add 액션</strong>이 기록됨</li><li>별도의 <strong>Deletion Vector 파일(.dv)</strong> 이 생성되어 삭제된 Row ID를 보관</li></ol><h4>🖼 동작 다이어그램</h4><pre>삭제 요청 →<br>  _delta_log JSON 업데이트<br>    ├── remove 액션 (기존 파일 무효화)<br>    └── add 액션 (DV 메타데이터 참조)<br>  ↓<br>  Deletion Vector 파일(.bin) 생성<br>    └── 삭제된 row 위치를 비트맵/리스트로 저장</pre><p>쿼리 엔진 동작 순서:</p><ol><li>_delta_log에서 최신 스냅샷(유효한 파일 목록) 확인</li><li>유효한 Parquet 파일 + 연결된 DV 메타데이터 로딩</li><li>DV 메타데이터에 지정된 .bin 파일 참조</li><li>삭제된 row를 제외하고 결과 반환</li></ol><h4>📂 실제 디렉토리 구조 예시 (S3)</h4><p>파티션이 설정된 델타 테이블기준으로 아래와 같은 디렉토리 구성됨</p><pre>tables/<br>└── /delta-table/                              # Delta Table Root<br>    ├── _delta_log/                            # 트랜잭션 로그<br>    │   ├── 00000000000000000010.json<br>    │   ├── 00000000000000000011.json<br>    │   └── 00000000000000000012.checkpoint.parquet<br>    │<br>    ├── 01/                                    # 파티션 디렉토리<br>    │   ├── part-00000-xxxx.snappy.parquet<br>    │   └── part-00001-xxxx.snappy.parquet<br>    │<br>    ├── deletion_vector_0014c02e-....bin       # DV 파일 (삭제된 row 기록)<br>    └── deletion_vector_0a7d33ab-....bin</pre><p><strong>_delta_log/</strong></p><ul><li>JSON + Checkpoint Parquet</li><li>어떤 파일이 추가/삭제되었는지, 그리고 DV 메타데이터를 기록</li></ul><p><strong>파티션 디렉토리 (예: 01/)</strong></p><ul><li>실제 데이터 파일(Parquet)이 저장되는 위치</li></ul><p><strong>deletion_vector_xxxx.bin</strong></p><ul><li>삭제된 row들의 위치(비트맵/리스트)를 저장하는 별도 바이너리 파일</li><li>_delta_log의 add 액션에서 이 파일을 참조 (pathOrInlineDv 필드)</li></ul><h4>📑 DV 메타데이터 예시</h4><p><strong>remove 액션</strong></p><p>_delta_log의 remove액션으로 기존 로그를 무효화합니다.</p><pre>{<br>  &quot;remove&quot;: {<br>    &quot;path&quot;: &quot;01/part-00000-54b2c529-...parquet&quot;,<br>    &quot;deletionTimestamp&quot;: 1684789501794,<br>    &quot;dataChange&quot;: true<br>  }<br>}</pre><p><strong>add 액션 (DV 정보 포함)</strong></p><p>_delta_log의 add 액션 안에서 DV 파일을 참조합니다.</p><pre>{<br>  &quot;add&quot;: {<br>    &quot;path&quot;: &quot;01/part-00000-54b2c529-...parquet&quot;,<br>    &quot;deletionVector&quot;: {<br>      &quot;storageType&quot;: &quot;u&quot;,<br>      &quot;pathOrInlineDv&quot;: &quot;deletion_vector_0014c02e-c8c8-498a-b9f4-ac4774167a7b.bin&quot;,<br>      &quot;offset&quot;: 85,<br>      &quot;sizeInBytes&quot;: 34,<br>      &quot;cardinality&quot;: 1<br>    }<br>  }<br>}</pre><ul><li><strong>pathOrInlineDv</strong>: DV가 저장된 파일 경로 (또는 인라인 데이터)</li><li><strong>sizeInBytes</strong>: DV 파일 크기</li><li><strong>cardinality</strong>: 삭제된 Row 수</li></ul><h3>4. DV의 장점과 주의점</h3><h4>장점</h4><ul><li>빠른 삭제/업데이트 처리</li><li>불필요한 Parquet 파일 재작성 최소화</li><li>쿼리 시점에만 삭제된 행을 제외</li></ul><h4>주의점</h4><ul><li>DV가 누적되면, 여전히 파일 재작성(Reorg, Optimize)이 필요</li><li>DV를 관리하는 메타데이터 크기가 커지면 성능 저하 가능</li><li>결국 장기적으로는 Vacuum/Optimize로 물리적 정리 필수</li></ul><h3>5. 운영 전략</h3><p>실제 운영 환경에서 DV를 관리하려면 다음이 필요합니다.</p><ol><li><strong>DV 모니터링</strong></li></ol><ul><li>DESCRIBE DETAIL로 DV 사용 여부 확인</li><li>일정 기준 이상이면 정리 작업 수행</li></ul><p><strong>2. Reorg / Optimize</strong></p><ul><li>DV가 많은 파일은 강제 재작성(Reorg)</li><li>Optimize(Z-Order, Bin-Packing)로 병합 및 정렬</li></ul><p><strong>3. Vacuum 실행</strong></p><ul><li>참조되지 않는 파일을 물리적으로 삭제</li><li>Retention 기간을 운영 정책에 맞게 조정</li></ul><p><strong>4. 스토리지 모니터링</strong></p><ul><li>DV만으로는 스토리지 용량이 바로 줄지 않음</li><li>Optimize &amp; Vacuum 후에야 실제 반영</li></ul><h3>6. 마무리</h3><p>Deletion Vector는 Delta Table이 대규모 분산 환경에서<br><strong>효율적인 삭제와 업데이트</strong>를 가능하게 하는 핵심 기술입니다.</p><p>하지만 DV가 쌓이기만 하면 성능이 떨어지므로,<br><strong>Reorg, Optimize, Vacuum</strong>과 함께 관리해야 합니다.</p><p>다음 글에서는 Delta Table의 <strong>Optimize 전략<br></strong>(Bin-Packing, Z-Order, Reorg, Vacuum)을 구체적으로 다루겠습니다.</p><h3>참고자료</h3><ul><li><a href="https://delta.io/blog/2023-07-05-deletion-vectors/">Delta Lake Deletion Vectors 공식 블로그</a></li><li><a href="https://docs.databricks.com/en/delta/deletion-vectors.html">Databricks Documentation: Deletion Vectors</a></li></ul><h4>💡 운영 인사이트</h4><blockquote><em>실제 운영 환경에서 우리는 </em><strong><em>매일 데이터를 삭제</em></strong><em>해야 하기 때문에 DV 관련 실험을 자주 해왔습니다.</em></blockquote><blockquote><em>그 과정에서 얻은 교훈을 정리하면 다음과 같습니다:</em></blockquote><blockquote><strong>(1) DV와 스토리지 용량<br> — </strong>DV가 누적돼도 S3의 실제 사용량은 바로 줄지 않습니다.<br> — 새로운 파일이 생성되어, 참조가 교체돼야 Vacuum 대상이 됩니다.<br> — 따라서 DV 모니터링 + Optimize/Vacuum 조합이 핵심입니다.</blockquote><blockquote><strong>(2) Optimize 전략<br> — </strong>Optimize를 전체 기간에 걸기보다, 특정 기간(eg. 최근 7일)으로 범위를 좁히면 병합이 더 잘되는 경향을 보였습니다.</blockquote><blockquote><strong>(3) DV 개수 모니터링<br> — </strong>DESCRIBE DETAIL &lt;table&gt;로 DV 개수를 확인할 수 있습니다.<br> — DV가 일정 기준 이상 쌓이면 Reorg나 Optimize를 트리거하는 자동화를 두는 것도 고려해볼만 한 것 같습니다.</blockquote>
