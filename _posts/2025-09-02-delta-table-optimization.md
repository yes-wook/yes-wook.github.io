<p>Delta Lake 기반 환경을 운영하다 보면 작은 파일 증가, DV 누적, 특정 조건 검색 지연 같은 문제가 반복됩니다.<br>이를 해결하기 위해 제공되는 대표 기능은 <strong>Optimize, Reorg, Z-Order, Bloom Filter, Liquid Clustering</strong>입니다.</p><h3>1) Optimize (Bin-Packing)</h3><h4><strong>개념</strong></h4><p>작은 파일을 병합하여 적정 크기의 파일로 만드는 과정입니다.<br>이 과정에서 DV(Deletion Vector)가 적용된 파일도 새로 작성되면서 정리 효과가 나타납니다.</p><h4><strong>특징</strong></h4><ul><li><strong>Small File Problem</strong> 해소</li><li>Bin-Packing으로 파일 크기 균일화 (권장 target size: 약 1GB )</li><li>Optimize는 새로운 데이터를 추가하는 작업이 아님 → 로그에 dataChange=false로 기록됨<br>| 즉, <strong>파일만 바뀌었을 뿐 데이터 내용은 동일<br></strong>| downstream 증분 파이프라인이 같은 데이터를 <strong>다시 처리하지 않음</strong></li><li>파티션 단위로 범위 지정 가능 → 불필요한 비용 방지</li></ul><h4><strong>그림</strong></h4><pre>## _before_<br>part-00011.parquet (50MB)<br>part-00012.parquet (60MB)<br>part-00013.parquet (45MB)<br><br>## _after_<br>part-00100.parquet (~1GB) ← 병합 + DV 정리</pre><p><strong>운영 팁</strong></p><ul><li>eg. 매일 새벽, 최근 7일만 Optimize → 비용 절감</li><li>eg. 매월 신규 파티션만 Optimize</li></ul><h3>2) Reorg (File Compaction)</h3><h4><strong>개념</strong></h4><p>DV가 누적된 특정 파일만 강제로 재작성하는 기능.</p><h4><strong>특징</strong></h4><ul><li>파일 단위 대상</li><li><strong>DV 정리</strong>에 집중</li><li><strong>Row Group 단위</strong></li></ul><h4><strong>그림</strong></h4><pre>## _before_<br>part-00001.parquet + DV(bin: g1, g2, g3)<br><br>## _after_<br>part-00021.parquet (DV 반영, 정리 완료)</pre><h4>운영 팁</h4><ul><li>DV 비율이 높은 파티션에만 선별 적용</li><li>비용이 크므로 전체 테이블 주기 실행은 지양</li></ul><h3>3) Z-Order Clustering</h3><h4><strong>개념</strong></h4><p>여러 컬럼을 기준으로 데이터를 물리적으로 근접하게 배치하는 기법.<br>Range/IN 조건 검색 성능을 크게 개선합니다.</p><h4><strong>특징</strong></h4><ul><li>Range/IN 조건 성능 향상</li><li>Equality 조건에서는 효과 제한적</li><li>Z-Order는 Optimize 실행 시 내부적으로 <strong>파일 병합 + 행 순서 재배치</strong></li><li>행 재배치는 <strong>Row Group 단위</strong></li></ul><h4>동작 방식</h4><p><strong>1/ 파티션 경계는 그대로 유지</strong></p><ul><li>Z-Order는 파티션 스키마(예: date_ym=&#39;2025-09&#39;)를 변경하지 않습니다.</li><li>파티션 디렉토리 구조는 그대로 남습니다.</li></ul><p><strong>2/ 파티션 내부 파일을 다시 씀</strong></p><ul><li>같은 파티션에 속한 여러 작은 파일들을 <strong>합치거나(bin-packing)</strong>,</li><li>큰 파일이라도 <strong>행 순서를 재배치(Z-curve)</strong>하면서 다시 씁니다.</li><li>즉, <em>“파일 재구성 + 행 재배치”</em> 단계가 동시 발생.</li></ul><p><strong>3/ Row Group 최적화</strong></p><ul><li>min/max 범위가 좁아져, file skipping강화됨</li></ul><h4><strong>그림</strong></h4><pre>## _before (date_ym = 2025-09/)<br>├─ part-00001.parquet (KR/US/DE 섞임)<br>├─ part-00002.parquet (KR/US/DE 섞임)<br>├─ part-00003.parquet (KR/US/DE 섞임)<br><br>## _after (date_ym = 2025-09/, ZORDER BY (country, user_id))<br>├─ part-00100.parquet<br>│   RowGroup1: KR 사용자 모음<br>│   RowGroup2: US 사용자 모음<br>│   RowGroup3: DE 사용자 모음<br>├─ part-00101.parquet<br>│   RowGroup1: KR 사용자 모음<br>│   RowGroup2: US 사용자 모음<br>│   RowGroup3: DE 사용자 모음</pre><h4>요약</h4><ul><li>Z-Order는 <strong>파티션을 건드리지 않는다</strong>.</li><li>파티션 내부에서 <strong>파일 병합 + 행(Row Group) 재배치</strong>를 수행한다.</li><li>따라서, 여러 파일을 <strong>재구성(합치거나 나누기)</strong> 하면서, 각 파일 안의 Row Group 순서를 최적화한다.</li></ul><h4><strong>운영 팁</strong></h4><ul><li>2~3개 주요 조회 조건 컬럼을 선정</li><li>Equality보다는 Range 필터에 효과</li><li>비용이 크므로, 핵심 테이블 위주로 사용</li></ul><h3>4) Bloom Filter Index</h3><h4><strong>개념</strong></h4><p>특정 값이 <strong>“존재하지 않음” </strong>을 빠르게 확인해, 불필요한 파일 읽기를 줄이는 확률적 인덱스.</p><h4><strong>특징</strong></h4><ul><li>High-cardinality 컬럼(user_id, device_id 등)에 적합</li><li>_delta_index/에 인덱스파일 생성됨</li><li>False Positive 확률(fpp) 조정 가능</li><li>Equality 조회 성능 최적화</li></ul><h4><strong>그림</strong></h4><pre>/table_root/<br>├─ _delta_log/<br>│   ├─ 00000000000000000010.json<br>│   └─ ...<br>├─ 00/                      ← 실제 데이터 파티션<br>    ├─ part-00030-xxxx.snappy.parquet<br>    └─ part-00031-xxxx.snappy.parquet<br>    └─_delta_index/            ← Bloom Filter 등 인덱스 파일 저장소<br>        ├─ part-00030-....index.v1.parquet<br>        ├─ part-00031-....index.v1.parquet<br>        └─ ...</pre><ul><li>_delta_log는 인덱스 생성·갱신 여부를 기록</li><li>_delta_index는 실제 인덱스 데이터를 담음</li></ul><h4><strong>쿼리 동작</strong></h4><pre>쿼리 실행 (WHERE user_id=123)<br>  ↓<br>_delta_index/*.parquet.index 조회<br>  ↓<br>File A: Bloom filter says &quot;123 없음&quot; → 스킵<br>File B: Bloom filter says &quot;123 있을 수 있음&quot; → 해당 파일만 스캔</pre><h4><strong>운영 팁</strong></h4><ul><li>user_id, device_id 같은 high-cardinality 컬럼에 적합</li><li>fpp=0.01~0.05 정도 설정 (정확도 vs 인덱스 크기 트레이드오프)</li><li>Optimize/Reorg로 데이터 파일이 바뀌면 인덱스도 재작성됨</li></ul><h3>5) Liquid Clustering</h3><h4><strong>개념</strong></h4><p>정적 파티션 대신 <strong>클러스터링 키 기준으로 파일을 자동 배치</strong>.<br>데이터가 새로 들어와도 지속적으로 <strong>클러스터링 상태를 유지</strong>.</p><h4><strong>특징</strong></h4><ul><li>파티션 설계 부담 감소</li><li>데이터 유입 패턴 변해도 자동 정렬 유지</li><li>Row Group 밀도가 높아져 Data Skipping 성능 확보</li></ul><h4><strong>그림</strong></h4><pre>## _before (혼재된 파일)_<br>part-00001.parquet<br>  RowGroup 1: KR, US, DE 섞임<br>  RowGroup 2: KR, US, DE 섞임<br><br>part-00002.parquet<br>  RowGroup 1: KR, US, DE 섞임<br>  RowGroup 2: KR, US, DE 섞임<br><br><br>## _after (CLUSTER BY (country,date_ym))_<br>part-01000.parquet<br>  RowGroup 1: DE-2025-08<br>  RowGroup 2: DE-2025-09<br><br>part-01001.parquet<br>  RowGroup 1: KR-2025-08<br>  RowGroup 2: KR-2025-09<br><br>part-01002.parquet<br>  RowGroup 1: US-2025-08<br>  RowGroup 2: US-2025-09</pre><ul><li><strong>Before</strong>: 파일과 Row Group 안에 여러 국가가 뒤섞임</li><li><strong>After</strong>: 파일 단위로도, Row Group 단위로도 “country+date_ym”가 더 순도 높게 모임</li></ul><h4><strong>운영 팁</strong></h4><ul><li>변동 큰 이벤트 로그 테이블에 우선 적용</li><li>기존 파티션 방식보다 유연성 높음</li></ul><h3>6) Vacuum</h3><h4><strong>개념</strong></h4><p>더 이상 참조되지 않는 파일을 실제 스토리지에서 삭제.</p><h4><strong>특징</strong></h4><ul><li><strong>스토리지 절감</strong>이 목적 (쿼리 성능에는 영향 없음)</li><li>Retention 기간 내 파일은 Time Travel 지원, Retention 초과 시 실제 삭제</li></ul><h4><strong>그림</strong></h4><pre>## _before_<br>part-00011.parquet (obsolete)<br>g1.bin (DV 파일)<br><br>## _after_<br>part-00110.parquet (현행만 유지)</pre><p><strong>운영 팁</strong></p><ul><li>Retention ≥ 7일 권장 (데이터 복구 안전성)</li><li>Weekly Vacuum으로 스토리지 회수</li></ul><h3>7) 요약 표</h3><pre>+------------------+--------------------------+-------------+--------------------------+--------------------+<br>| 기능             | 목적                      | 범위         | 파일 변화                 | 운영 팁             |<br>+------------------+--------------------------+-------------+--------------------------+--------------------+<br>| Optimize         | Small File 병합 + DV 정리 | 지정 범위     | 작은 파일 → 큰 파일 병합    | 최근 7~14일 실행     |<br>| Reorg            | DV 누적 파일 정리          | 특정 파일      | DV 포함 파일 → 재작성      | DV 비율 높을 때      |<br>| Z-Order          | 다차원 조건 검색 최적화      | 컬럼 단위      | 데이터 물리적 재배치        | Range 필터 유리     |<br>| Bloom Filter     | 값 존재 여부 빠른 판정       | 컬럼 단위      | _delta_index 인덱스 파일  | high-card 컬럼 적용  |<br>| Liquid Clustering| 자동 클러스터링, 파티션 대체  | 전체 테이블    | 동적 클러스터 디렉토리      | 변동 큰 테이블 적용   |<br>| Vacuum           | obsolete 파일 물리 삭제   | 전체 테이블    | 불필요 파일 삭제           | Retain ≥ 7일 권장   |<br>+------------------+--------------------------+-------------+--------------------------+--------------------+</pre><h3>마무리</h3><p>Delta Lake는 단순히 데이터를 저장하는 레이어가 아니라, <strong>운영 전략을 어떻게 짜느냐에 따라 비용·성능·유연성이 크게 달라지는 관리 레이어</strong>입니다.</p><ul><li><strong>쿼리 성능이 문제면 → Optimize + Z-Order</strong></li><li><strong>삭제/갱신이 많은 테이블은 → Reorg 주기 관리</strong></li><li><strong>사용자 ID·디바이스 ID 검색이 많으면 → Bloom Filter</strong></li><li><strong>새로운 데이터 모델링이라면 → Liquid Clustering 고려</strong></li><li><strong>스토리지 절감이 필요하다면 → Vacuum 주기 설계</strong></li></ul><p>즉, 모든 기능을 “무조건 다 돌린다”가 아니라, <strong>테이블 특성과 워크로드에 맞는 조합</strong>이 핵심입니다.</p><p>저희 팀에서도 현재 수십 테라바이트 단위 로그 테이블에서는 “Optimize+Bloom Filter”만 쓰고, DV를 모니터링 하며, reorg를 줄이는 방식으로 <strong>최적화 전략을 따로 가져가고 있습니다.</strong></p><p>참고:</p><ul><li><a AWS</a></li></ul><p><a href="https://docs.databricks.com/aws/en/delta/clustering">https://docs.databricks.com/aws/en/delta/clustering</a></p>
