## TL;DR

대용량 Delta 테이블에서 Bloom Filter는 모든 조회를 빠르게 만드는 만능 인덱스가 아니다. 나는 소수 식별자 조회에서 효과를 확인해 적용했지만, 이후 Photon과 Predictive I/O 환경에서 같은 workload를 다시 측정한 뒤 제거했다. 핵심은 특정 기능을 고정하는 것이 아니라 **workload·쓰기 비용·런타임 동작이 바뀔 때마다 선택을 재검증하는 것**이다.

## 1. 왜 Bloom Filter를 검토했나

대규모 로그 테이블은 시간 단위로 유입되고, 분석 시에는 특정 사용자·기기 식별자를 찾아야 하는 경우가 있다. 날짜 파티션만으로는 고카디널리티 조건의 파일 스킵이 충분하지 않을 수 있다.

Bloom Filter는 행을 직접 줄이는 기능이 아니라, 조건 값이 없을 가능성이 높은 **파일을 읽기 전에 제외하는 file pruning 보조 수단**이다. 그래서 “특정 키를 몇 개 찾는가”와 “이미 파티션·통계로 얼마나 걸러지는가”가 효과를 좌우한다.

## 2. 비교한 선택지

| 선택지 | 기대 효과 | 비용·주의점 |
|---|---|---|
| 기본 통계·파티션 | 단순하고 유지비가 낮은 file pruning | 고카디널리티 조건에서는 한계 |
| Bloom Filter | 소수 키 point lookup의 파일 스킵 보완 | 인덱스 생성·유지와 계획 비용 |
| Z-ORDER | 관련 값을 파일 단위로 모아 skipping 강화 | 재정렬 비용이 높고 쓰기 workload에 영향 |
| REORG | Deletion Vector 물질화·파일 정리 | 실행 시 읽기·쓰기 비용 발생 |
| Photon·Predictive I/O | 엔진이 조건과 파일을 더 효율적으로 판단 | 런타임·테이블 상태에 따라 재검증 필요 |

## 3. 첫 번째 판단: workload를 나눠 측정하기

소수 키 point lookup에서는 Bloom Filter가 실행시간을 줄였고, Z-ORDER를 함께 적용하면 더 큰 개선을 관찰했다. 반면 다수 키를 한 번에 조회하거나 JOIN을 수행하는 workload에서는 효과가 제한적이었다.

여기서 중요한 것은 “가장 빠른 옵션”이 아니라 “반복되는 workload에 비용을 정당화할 수 있는 옵션”이었다. 재정렬 비용이 높은 Z-ORDER는 낮은 조회 빈도에 비해 부담이 컸고, 그래서 상대적으로 가벼운 Bloom Filter만 운영에 적용했다.

## 4. 두 번째 판단: 기능은 영구 설정이 아니다

1년 뒤 런타임과 I/O 최적화 기능이 달라진 뒤 같은 조건을 다시 확인했다. 이때 Bloom Filter는 기대한 추가 skipping보다 인덱스 유지와 실행 경로 측면의 부담이 커질 수 있었다. 인덱스를 제거한 뒤 Photon 기반 실행 경로가 더 잘 활용되고, 소수 키 조회 시간도 개선되는 것을 확인했다.

이 경험은 “한때 유효했던 최적화”가 “계속 유효한 최적화”는 아니라는 점을 보여준다. Databricks도 Bloom Filter 대신 최신 데이터 스키핑·클러스터링 전략을 검토하도록 권고하고 있다.

## 5. 운영 원칙

1. **workload를 먼저 분류한다.** point lookup, 다건 필터, JOIN, 삭제·적재를 분리해 본다.
2. **읽기와 쓰기 비용을 함께 본다.** 조회 개선만 보고 재정렬·인덱스 유지 비용을 빼면 안 된다.
3. **파일 수와 테이블 상태를 관찰한다.** OPTIMIZE, Deletion Vector, 파일 크기, 파티션을 함께 본다.
4. **재검증 시점을 정한다.** 런타임 업그레이드, 데이터 증가, 조회 패턴 변화가 발생하면 다시 측정한다.
5. **제거 기준도 문서화한다.** 기능을 끄는 판단 역시 성능·비용·운영 복잡도에 근거해야 한다.

## 마무리

Delta Lake 최적화에서 중요한 것은 Bloom Filter나 Z-ORDER 자체가 아니다. 현재 workload에서 어떤 file skipping 전략이 필요한지, 그 효과가 쓰기·유지보수 비용을 넘는지, 그리고 엔진이 바뀐 뒤에도 같은 판단이 유효한지를 지속적으로 확인하는 일이다.

*관련 공식 문서와 공개 가능한 재현 예제는 후속 글에서 연결할 예정이다.*
